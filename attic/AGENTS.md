# attic (crate)

Shared library for the Attic Nix binary cache project. Provides the data
types, wire-format API structs, hashing/signing primitives, and (optionally)
FFI bindings to `libnixstore` used by `attic-client`, `attic-server`, and
`attic-token`. No network/HTTP client or server code lives here — this is a
"model + protocol + utility" crate, not a service.

## Module map

- `lib.rs` — crate root; `#![deny(unsafe_code, ...)]` at the crate level
  (unsafe is only permitted where explicitly `#[allow(unsafe_code)]`'d, e.g.
  FFI glue and a few documented invariants).
- `cache.rs` — `CacheName` / `CacheNamePattern`: validated, serde-aware cache
  identifiers and glob-style patterns used for ACLs.
- `error.rs` — `AtticError`/`AtticResult`, the crate-wide error enum; `From`
  impls bridge `io::Error`, `signing::Error`, `hash::Error`, and (feature
  `nix_store`) `cxx::Exception`.
- `hash/mod.rs` — `Hash` enum (currently SHA-256 only), Nix "typed" hash
  string parsing/formatting (`sha256:<base16|base32>`).
- `mime.rs` — MIME-type constants for `/nix-cache-info`, `.narinfo`, `.nar`.
- `chunking/mod.rs` (feature `chunking`) — FastCDC-based content-defined
  chunking of an `AsyncRead` NAR stream into `Bytes` chunks.
- `io/mod.rs` + `io/hash_reader.rs` (feature `io`) — `merge_chunks` (lazily
  reassembles/prefetches chunk streams, e.g. from S3), `read_chunk_async`,
  and `HashReader` (an `AsyncRead`/`AsyncBufRead` wrapper that hashes bytes
  as they're read and finalizes on EOF).
- `nix_store/mod.rs` — `StorePath`/`StorePathHash`/`ValidPathInfo` and store
  path/base-name validation; these pure-Rust types exist regardless of the
  `nix_store` feature.
  - `nix_store/store.rs` (feature `nix_store`) — `NixStore`, the high-level
    async wrapper (`connect`, `parse_store_path`, `follow_store_path`,
    `nar_from_path`, `compute_fs_closure[_multi]`, `query_path_info`).
  - `nix_store/bindings/mod.rs` + `nix.cpp`/`nix.hpp` (feature `nix_store`)
    — the `cxx`-generated FFI bridge (`CNixStore`, `CPathInfo`) plus the
    `AsyncWriteAdapter`/`AsyncWriteSender` channel used to stream NAR bytes
    out of blocking C++ calls into async Rust.
- `signing/mod.rs` — `NixKeypair`/`NixPublicKey` (ed25519), matching Nix's
  `{keyName}:{base64Payload}` canonical string format; keys generated by the
  Nix CLI work here unmodified.
- `api/` — wire types shared between client and server for the Attic HTTP
  API: `api/binary_cache.rs` (visibility header), `api/v1/cache_config.rs`
  (create/list cache), `api/v1/get_missing_paths.rs`, `api/v1/upload_path.rs`
  (`UploadPathNarInfo`, upload headers/result types).
- `testing/mod.rs` (all platforms) — `get_runtime()`, `get_fake_data()` test
  helpers.
  - `testing/shadow_store/mod.rs` (`cfg(target_family = "unix")`) —
    `ShadowStore`, a temp Nix store root + `nix-store` wrapper script for
    integration tests that talk to a real Attic server via vanilla Nix.
- `util.rs` (feature `tokio`) — `Finally<F>`, a guard that spawns a future
  when dropped (cleanup for resources that don't fit ownership models).
- `build.rs` — only active under feature `nix_store`; compiles the C++ side
  of the FFI bridge via `cxx_build` + `system_deps`/`pkg-config`.

## Cargo features

- `default = ["chunking", "io", "nix_store", "tokio"]`.
- `tokio` — pulls in `tokio` with `rt`+`time`; base requirement for almost
  everything async in this crate.
- `io` — chunk-merging/hashing stream utilities (`io` module); implies
  `tokio` + `async-stream`. Used by both client (assembling downloads) and
  server (chunk storage).
- `chunking` — FastCDC splitting; implies `tokio` + `io`. Used by client
  (push) and server (dedup storage).
- `nix_store` — enables the real `libnixstore` FFI (`cxx`, `cxx-build`,
  `system-deps`, plus `tokio/fs` + `tokio/process`). Without it, `nix_store`
  still compiles but only exposes the pure-Rust `StorePath`/`StorePathHash`
  validation types — `NixStore` (the FFI-backed store handle) does not
  exist. Only the **client** needs this: per `nix_store/mod.rs`, "we never
  interact with the Nix store on the server."

Consumer feature selection (from each crate's `Cargo.toml`):
- `client` — default features (needs `nix_store` for real store access).
- `server` — `default-features = false, features = ["chunking", "io",
  "tokio"]` — deliberately **excludes** `nix_store` (and thus avoids linking
  `libnixstore`/needing a C++ toolchain or `pkg-config` nix dev packages on
  the server build).
- `token` — `default-features = false` (no extra features) — only needs the
  plain data types (`CacheName`, `CacheNamePattern`); this lets `attic-token`
  build for `wasm32-unknown-unknown` (see `justfile`'s `ci-build-wasm`,
  which also builds `attic` itself as `--no-default-features -F chunking -F
  io` for wasm, i.e. still without `nix_store`).

## FFI / build notes

- `build.rs` runs `cxx_build::bridge("src/nix_store/bindings/mod.rs")`,
  compiles `src/nix_store/bindings/nix.cpp` with `-std=c++23 -O2`, and links
  against `libnixstore`/`libnixmain` located via `system-deps` (see
  `[package.metadata.system-deps]`: `nix-store`/`nix-main` version `2.24`,
  gated on the `nix_store` feature). This requires `pkg-config` and Nix's
  C++ dev headers/libs to be on the system (or in the Nix dev shell —
  `shell.nix`/`crane.nix` provide `pkg-config` and Nix).
- The build script defines `ATTIC_VARIANT_NIX` and a `NIX_VERSION` macro
  (`major*100 + minor`) derived from the detected `nix-main` version, and
  panics if `minor >= 100` (unsupported Nix version scheme).
- All FFI code is `#[allow(unsafe_code)]`'d explicitly; the rest of the
  crate denies `unsafe_code` at the crate level (`lib.rs`).
- `AsyncWriteAdapter`/`AsyncWriteSender` bridge blocking C++ NAR streaming
  (`spawn_blocking`) into an async `Stream`/`AsyncWrite` on the Rust side —
  don't call FFI methods directly from an async context; go through
  `spawn_blocking` as `store.rs` does.

## Key invariants & conventions

- **Cache names** (`cache.rs`): ASCII alphanumeric + `-_+`, ≤50 chars, must
  start with an alphanumeric char. `+` is the namespace/user-name delimiter
  convention (e.g. `zhaofengli+shared`). `CacheNamePattern` additionally
  allows `*` wildcards and is only meant to be produced by trusted entities
  (e.g. JWT claims), never arbitrary user input.
- **Store paths** (`nix_store/mod.rs`): base name = 32-char Nix base32 hash
  (`STORE_PATH_HASH_LEN`, alphabet excludes `e/o/u/t`) + `-` + human-readable
  name (`[A-Za-z0-9+-._?=]+`). `StorePath::from_base_name_unchecked` /
  `StorePathHash::new_unchecked` are `unsafe` escape hatches — only use them
  when a value has already been validated by the C++ side or another trusted
  source.
- **Hashes** (`hash/mod.rs`): canonical on-the-wire form is
  `to_typed_base16()` ("the canonical representation of hashes in the Attic
  database"); Nix-native form is `to_typed_base32()`. Only SHA-256 is
  currently modeled.
- **Signing** (`signing/mod.rs`): canonical string format is always
  `{keyName}:{base64Payload}`, matching Nix CLI-generated keys exactly — do
  not change this format casually, it's a cross-compat contract with Nix.
- **Chunking** (`chunking/mod.rs`): content-defined chunking via FastCDC
  (`ronomon` variant) over uncompressed NARs; chunk boundaries are
  content-dependent, not size-dependent, which is what makes cross-upload
  deduplication possible on the server. Don't assume fixed-size chunks in
  tests or storage logic.
- `HashReader` finalizes its digest only at EOF (empty read / empty
  `fill_buf`); mixing `AsyncRead::read` and `AsyncBufRead::fill_buf`/
  `consume` on the same instance is supported and tested.

## Building / testing this crate

- Default build/test needs a Nix installation with `libnixstore`/
  `libnixmain` dev files discoverable via `pkg-config` (version ~2.24) and a
  C++23-capable compiler, because `nix_store` is a default feature:
  `cargo test -p attic`. Easiest inside the project's Nix dev shell
  (`shell.nix` / `nix develop`), which wires up `pkg-config`, Nix, and the
  compiler.
- To build/test without the C++ FFI dependency (pure-Rust subset only):
  `cargo test -p attic --no-default-features --features chunking,io,tokio`.
- Benchmarks: `cargo bench -p attic` (criterion; see `[[bench]] name =
  "chunking"`, benches live under `attic/benches/` if present).
- `nix_store` integration tests (`src/nix_store/tests/`) use recorded NAR
  fixtures under `src/nix_store/tests/nar/` and real `.drv` files under
  `src/nix_store/tests/drv/` — see `src/nix_store/tests/README.md` before
  touching these.
- CI equivalents (see `justfile`): `just ci-unit-tests <matrix>` (runs
  through `nix build`), `just ci-build-wasm` (builds `attic` and
  `attic-token` for `wasm32-unknown-unknown` with `nix_store` disabled).

## Consumers

- `client` (`attic-client`, path `../client`) — default features; the only
  consumer of the FFI-backed `NixStore` (`nix_store` feature), since the
  client runs on developer/CI machines with a real Nix install.
- `server` (`attic-server` / `atticd` / `atticadm`, path `../server`) —
  `chunking` + `io` + `tokio` only, no `nix_store`; the server manipulates
  NARs/chunks/hashes but never calls into `libnixstore` directly.
- `token` (`attic-token`, path `../token`) — no extra features; only uses
  `cache::{CacheName, CacheNamePattern}` for JWT permission claims. Must stay
  buildable on `wasm32-unknown-unknown`, so avoid adding new default-feature
  dependencies here that aren't wasm-safe.

## Gotchas

- `pub mod nix_store;` in `lib.rs` is **unconditional** — the module always
  exists, but `store::NixStore` and the `bindings` submodule only exist
  under `#[cfg(feature = "nix_store")]`. Code referencing `NixStore` from
  outside this crate must be feature-gated the same way (see how `client`
  does it).
- Several functions in `nix_store/mod.rs` (`from_base_name`, `to_base_name`,
  `as_base_name_bytes`) are marked
  `#[cfg_attr(not(feature = "nix_store"), allow(dead_code))]` because
  they're currently only consumed by the FFI-backed code path — if you add a
  new pure-Rust consumer, this warning suppression may need to move.
- `to_hash()`/`name()`/`as_base_name_bytes()` are additionally
  `#[cfg(target_family = "unix")]` (byte-level `OsStr` manipulation), so this
  crate cannot fully support non-Unix targets for those APIs.
  `testing/shadow_store` is Unix-only for the same reason (`PermissionsExt`).
- `wasm32-unknown-unknown` builds only work with `nix_store` disabled (it
  needs the C++ toolchain/`libnixstore` dev packages); don't add it to
  `server` or `token`'s feature sets without a strong reason.
- The `api/` module types are the client/server wire format — changing field
  names/types in `UploadPathNarInfo`, `CreateCacheRequest`, etc. is a
  breaking API change across client/server.
- The crate denies `unsafe_code` globally; any new `unsafe` needs an
  explicit `#[allow(unsafe_code)]` and should be scoped as tightly as
  possible (follow the pattern in `nix_store/mod.rs` and `bindings/mod.rs`).
