# Upload Reliability and Diagnostics

## Interpreting HTTP 499

HTTP 499 is normally generated by a reverse proxy after the downstream client
has closed its connection. It does not prove that Attic rejected the upload or
that the server stopped processing it. Correlate the proxy event with Attic's
`request_id` and upload lifecycle logs before assigning a cause.

The v1 `/_api/v1/upload-path` endpoint supports replaying a complete path
upload. The client retries transport failures and HTTP 408, 429, 499, 500,
502, 503, and 504 up to three total attempts, with full-jitter backoff starting
at 500 ms and capped at 5 seconds. It honors a valid `Retry-After` value within
that cap. Authentication, authorization, malformed input, and other 4xx
responses are not retried.

Replay is not resumability. Each retry generates and sends a complete NAR
stream; Attic has no public resumable-session or partial-upload protocol in
v1. A response lost after a committed upload can therefore be replayed safely,
but clients cannot resume from an arbitrary byte or chunk offset.

## Correlation and safe logging

Attic accepts a valid UUID in `X-Request-ID`, otherwise generates one, and
returns the chosen UUID in every application response, including rejected
requests and handler errors.
Pass that value through the proxy and use it to correlate client, proxy, and
server events.

TLS negotiation, connection failures, and failures that occur before Axum
receives a request cannot receive an application request ID. Use proxy and
load-balancer connection logs for those cases.

Upload logs record cache name, store path hash, declared and received byte
counts, chunk count, permit wait, timing, cleanup messages, storage/multipart
outcomes, and status. They intentionally do not log bearer tokens, request
bodies, local store paths, or other path metadata. Keep proxy logs to the same
standard when forwarding request IDs.

## Concurrency settings

The server configuration has global limits under `[uploads]`:

```toml
[uploads]
concurrent-uploads = 16
concurrent-chunk-uploads = 10
```

Both values must be greater than zero. The whole-upload limit is acquired only
after authentication. Chunk uploads retain the request-local bound and also use
the global chunk limit.

For S3-compatible storage, `multipart-concurrency` bounds in-flight multipart
parts and buffered read-ahead bodies:

```toml
[storage]
type = "s3"
multipart-concurrency = 4
```

Tune these values with storage request limits, CPU available for compression,
and memory available for buffered chunks. Increasing them without measuring
proxy and S3 behavior can make disconnect-related failures worse.

## Coverage and operational checks

Automated coverage verifies retry classification, attempt limits, content
length calculation, request-ID validation and response headers, upload-limit
validation, and multipart-concurrency validation. The retry tests use
synthetic transport and status errors only. They do not create a real network
disconnect, reconstruct a NAR body, or prove a lost response after a server
commit. Those scenarios remain deferred until a proxy/S3 failure-injection
environment exists.

Before a production rollout, deliberately interrupt an upload through the
actual proxy, inspect the matching `X-Request-ID` across proxy and Attic logs,
and verify that replay completes without leaked pending uploads or multipart
sessions. Repeat this after proxy timeout, buffering, or S3 policy changes.
