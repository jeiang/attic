# token (`attic-token`)

## Purpose

Implements Attic's access-control token format: parsing/extracting a JWT from
an HTTP `Authorization` header, verifying its signature (HS256 or RS256), and
answering "what can this bearer do to cache X" queries. Access control in
Attic is stateless — the server holds no session, it just validates the JWT
against a configured key on every request. The token format is intentionally
**opaque to clients**: only `sub`/`exp` and the namespaced claim below are
meaningful; nothing else should be interpreted by consumers.

Source: `src/lib.rs` (all logic), `src/util.rs` (Authorization header parsing),
`src/tests.rs` (fixture-based round-trip tests).

## Token/claim structure

Standard RFC 7519 claims (`sub`, `exp`, `nbf`/`iat`) plus one custom
namespaced claim, `https://jwt.attic.rs/v1` (`CLAIM_NAMESPACE` const),
required to be namespaced so platforms like Auth0 don't silently drop it.

Example payload:

```json
{
  "sub": "meow",
  "exp": 4102324986,
  "https://jwt.attic.rs/v1": {
    "caches": {
      "cache-rw": { "r": 1, "w": 1 },
      "cache-ro": { "r": 1 },
      "team-*":   { "r": 1, "w": 1, "cc": 1 }
    }
  }
}
```

`caches` is an `IndexMap<CacheNamePattern, CachePermission>` — keys are
patterns (may contain `*` wildcards), values are single-letter bit flags
serialized as `0`/`1` (via `serde_with::BoolFromInt`) to keep tokens small.
Absent flags default to `false` and are omitted from serialization
(`skip_serializing_if`).

There's also a separate `OidcToken` type (`OidcToken` struct) for verifying
raw OIDC ID tokens (RS256 only, requires issuer+audience match). It carries
no Attic namespace claim — its custom claims are kept as opaque `JsonValue`
for the server to apply its own OIDC-based authorization rules.

## Permissions

`CachePermission` fields (JSON key in parens):

- `pull` (`r`) — read/fetch objects from the cache
- `push` (`w`) — upload objects to the cache
- `delete` (`d`) — delete objects from the cache
- `create_cache` (`cc`) — create the cache itself (e.g. autonomously under a namespace like `team-*`)
- `configure_cache` (`cr`) — reconfigure cache settings
- `configure_cache_retention` (`cq`) — configure retention/quota settings
- `destroy_cache` (`cd`) — delete the cache itself

Each has a `require_*` method (generated by the `require_permission_function!`
macro) returning `Result<()>`. `can_discover()` is true iff *any* bit is set.
This matters for the error returned on failure:

- If the bearer has **no** permission at all on the cache → `Error::NoDiscoveryPermission` (server should return a generic 401, hiding whether the cache exists).
- If the bearer has **some** permission but not the one required → `Error::PermissionDenied` (server can return 403/404 as appropriate, since existence is already "discoverable").

`CachePermission::add_public_permissions()` grants implicit `pull` for caches
flagged public by the server — this is applied by the caller (server), not
by anything in this crate.

## Pattern matching semantics

Cache names (`attic::cache::CacheName`) are ASCII alnum/`-`/`_`/`+`, ≤50
chars, must start alphanumeric. `CacheNamePattern` allows `*` wildcards
(via the `wildmatch` crate) using the same charset otherwise.

`Token::get_permission_for_cache(cache)`:

1. Looks for an **exact** pattern match first (`cache.to_pattern()` — a
   pattern with no wildcard matcher, compared by string equality).
2. Otherwise iterates `caches` **in insertion/serialization order** and
   returns the **first** pattern whose `matches()` succeeds — not the most
   specific, not a merge of all matches.
3. If nothing matches, returns `CachePermission::default()` (all `false`,
   `can_discover() == false`).

Because of (2), **claim order in the JWT is semantically significant** —
a broader pattern listed before a narrower one will shadow it. `tests.rs`
(`test_basic`) specifically stress-tests this 1000x per token to guard
against `IndexMap`/HashSet-driven nondeterminism causing flaky auth
(this bit them before — see the `NOTE(cole-h)` comments).

## Key / signing details

`SignatureType` enum, all backed by `jwt-simple`:

- `HS256(HS256Key)` — symmetric HMAC; can sign and verify.
- `RS256(Box<RS256KeyPair>)` — asymmetric RSA; can sign and verify.
- `RS256PubkeyOnly(RS256PublicKey)` — verify only; `Token::encode` returns
  `Error::PubkeyOnlyCannotCreateToken` if you try to sign with it.

Helper constructors (all take a base64-encoded string, matching the
`ATTIC_SERVER_TOKEN_{HS256_SECRET,RS256_SECRET,RS256_PUBKEY}_BASE64` env vars
read by the server config):

- `decode_token_hs256_secret_base64` — base64 → raw HMAC secret bytes.
- `decode_token_rs256_secret_base64` — base64 → UTF-8 PEM (PKCS1) RSA private key.
- `decode_token_rs256_pubkey_base64` — base64 → UTF-8 PEM RSA public key.

`Token::from_jwt` verifies via `jwt_simple::VerificationOptions`, optionally
constraining `allowed_issuers`/`allowed_audiences` if the server is
configured with a bound issuer/audience. `Token::new` builds a fresh token
with only `sub`+`exp` set (and `nbf` = now); permissions are then populated
via `get_or_insert_permission_mut(pattern)` before `encode()`.

## How to test

```
cargo test -p attic-token
```

(Package name is `attic-token`, confirmed in `token/Cargo.toml`.) Tests live
in `src/tests.rs` (JWT round-trip + permission-matching fixtures, both
HS256 and RS256) and `src/util.rs` (Authorization header parsing).

**Known environment gotcha**: as of this writing, `cargo build -p attic-token`
fails in a sandbox without a `tokio` feature reaching `attic`, because
`attic/src/lib.rs` gates `pub mod testing;` on `#[cfg(target_family = "unix")]`
only, but `attic/src/testing/mod.rs` unconditionally calls `tokio::runtime`.
Since `token/Cargo.toml` depends on `attic` with `default-features = false`
(no `tokio` feature), this module fails to compile on any unix host. This is
a bug in the `attic` crate, not in `token` — do not "fix" it by adding
features to `token/Cargo.toml`; the correct fix is gating `testing` on
`feature = "tokio"` too.

## Consumers

Only `server` (`attic-server`, `../server`) depends on this crate directly:

- `server/src/access/mod.rs` does `pub use attic_token::*;` — the server's
  entire access-control API *is* this crate, plus an HTTP middleware layer.
- `server/src/access/http.rs` extracts the token from the `Authorization`
  header (via `util::parse_authorization_header`, supporting both `Bearer`
  and HTTP Basic where the password field is the JWT — used for Nix's
  `netrc`), verifies it with `Token::from_jwt` using the `SignatureType`
  built from server config, and stores it in per-request `AuthState`.
- `server/src/config.rs` builds `SignatureType`/`JWTSigningConfig` from the
  `ATTIC_SERVER_TOKEN_*_BASE64` env vars using this crate's decode helpers.
- The `atticadm` binary (also in `server`, see `server/Cargo.toml` `[[bin]]`)
  uses `Token::new`/`get_or_insert_permission_mut`/`encode` to mint tokens
  (`atticadm make-token`).

## Gotchas

- **Claim format is unstable/opaque by design** (see the module doc comment
  in `lib.rs`) — don't add client-side logic that parses or depends on the
  shape of the `https://jwt.attic.rs/v1` claim; only the server should ever
  interpret it. If you change `CachePermission`'s field set or rename tags,
  it breaks every previously-issued token for removed/renamed fields (there's
  no version field in the claim), so treat rename/removal of the short JSON
  keys (`r`, `w`, `d`, `cc`, `cr`, `cq`, `cd`) as a breaking change.
- Pattern iteration order matters for correctness (see above) — when adding
  permissions programmatically, be deliberate about insertion order.
- `CachePermission::default()` (no matching pattern) is indistinguishable
  from an explicitly-granted all-`false` entry — both yield
  `can_discover() == false`.
